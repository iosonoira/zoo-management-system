Zoo Management System — Roadmap iniziale

Dalla cartella vuota al primo servizio funzionante end-to-end · animal-service

Setup del mono-repo Maven
Obiettivo: struttura pulita, parent POM, Quarkus funzionante — prima di scrivere una riga di business logic
Crea la struttura di directory
Una cartella radice, dentro le 4 sottocartelle per i servizi + infrastructure + docs. Niente di più.
mkdir -p zoo-management-system/{animal-service,health-service,feeding-service,notification-service,infrastructure/keycloak,docs} cd zoo-management-system git init
Parent POM chiave
Il pom.xml nella radice gestisce le versioni per tutti i moduli. Tutti i figli ereditano da qui — mai duplicare versioni tra servizi.
<!-- zoo-management-system/pom.xml --> <groupId>it.zoo</groupId> <artifactId>zoo-management-system</artifactId> <packaging>pom</packaging> <modules> <module>animal-service</module> </modules> <properties> <quarkus.platform.version>3.20.0</quarkus.platform.version> <java.version>21</java.version> </properties>
Genera animal-service con Quarkus CLI
Non scrivere il pom figlio a mano — usa il CLI o code.quarkus.io. Seleziona solo le estensioni che servono subito.
quarkus create app it.zoo:animal-service \ --extension="rest-jackson,hibernate-orm-panache,jdbc-postgresql,flyway,oidc,smallrye-openapi" \ --no-code # --no-code: niente file di esempio da pulire dopo
Struttura dei package Java
Crea subito la struttura esagonale — anche vuota. Avere i package chiari fin dall'inizio evita di ritrovarsi tutto in un unico package dopo.
it.zoo.animal/ ├── domain/ │ ├── model/ ← Animal.java, AnimalStatus.java │ ├── port/ │ │ ├── in/ ← use case interfaces │ │ └── out/ ← repository interfaces │ └── exception/ ← AnimalNotFoundException.java ├── application/ ← use case implementations └── infrastructure/ ├── persistence/ ← JPA adapter ├── rest/ ← REST resource └── event/ ← Kafka producer
⚠ Non aggiungere Kafka e OIDC al pom subito. Aggiungi le estensioni solo quando arrivi a quel layer — ogni estensione inutilizzata aggiunge configurazione e può rompere lo startup.
Infrastruttura locale con Docker Compose
Obiettivo: PostgreSQL pronto prima di scrivere il primo adapter
docker-compose.yml minimale solo l'essenziale
In questa fase serve solo PostgreSQL per animal-service. Kafka e Keycloak li aggiungi più avanti. Un compose che parte senza errori vale più di uno completo che non si avvia.
# infrastructure/docker-compose.yml services: postgres-animal: image: postgres:16-alpine environment: POSTGRES_DB: animal_db POSTGRES_USER: zoo POSTGRES_PASSWORD: zoo_secret ports: ["5432:5432"] volumes: ["animal_data:/var/lib/postgresql/data"] volumes: animal_data:
application.properties per il dev profile
Configura il datasource puntando al container locale. Il dev profile di Quarkus usa %dev. come prefisso — non toccare il profile di produzione.
%dev.quarkus.datasource.db-kind=postgresql %dev.quarkus.datasource.username=zoo %dev.quarkus.datasource.password=zoo_secret %dev.quarkus.datasource.jdbc.url=jdbc:postgresql://localhost:5432/animal_db %dev.quarkus.hibernate-orm.database.generation=none %dev.quarkus.flyway.migrate-at-start=true
Verifica lo startup
Prima di toccare il domain, verifica che Quarkus parta senza errori contro il DB.
docker compose up -d # avvia postgres cd animal-service mvn quarkus:dev # deve partire senza eccezioni # atteso: "Listening on: http://localhost:8080"
✓ Checkpoint: Quarkus parte, si connette al DB, Swagger UI risponde su /q/swagger-ui. Solo allora vai al domain layer.
Domain layer — zero dipendenze framework
Obiettivo: modellare Animal, i port interfaces e le eccezioni. Nessuna annotazione Quarkus, Hibernate o Jakarta.
Animal — il model di dominio regola d'oro
Un POJO puro. Zero @Entity, zero @Column, zero @JsonProperty. Il domain non sa che esiste JPA né REST. Se vedi un'annotazione framework qui, è una violazione dell'architettura esagonale.
// domain/model/Animal.java public class Animal { private UUID id; private String name; private String species; private AnimalStatus status; private LocalDate arrivalDate; // costruttori, getter — nessuna annotazione }
Port interfaces — in e out
Le porte definiscono cosa il domain espone (in/) e di cosa ha bisogno (out/). Sono interfacce Java pure — le implementazioni stanno nell'infrastructure layer.
// port/in/RegisterAnimalUseCase.java public interface RegisterAnimalUseCase { Animal register(RegisterAnimalCommand command); } // port/out/AnimalRepository.java public interface AnimalRepository { Animal save(Animal animal); Optional<Animal> findById(UUID id); List<Animal> findAll(); }
Eccezioni di dominio
Estendono RuntimeException e vivono nel domain. Non contengono codici HTTP — quello è responsabilità dell'infrastructure layer.
// domain/exception/AnimalNotFoundException.java public class AnimalNotFoundException extends RuntimeException { public AnimalNotFoundException(UUID id) { super("Animal not found: " + id); } }
⚠ Test prima di andare avanti. Il domain layer si testa con JUnit puro — nessun @QuarkusTest, nessun mock di Quarkus. Se non riesci a testarlo con new Animal(), stai violando l'isolamento.
Application layer — i Use Case
Obiettivo: implementare i port "in" usando i port "out". Solo @ApplicationScoped e @Transactional — nient'altro di Quarkus.
Una classe = un use case regola
Ogni operazione di business ha la sua classe. RegisterAnimalService implementa RegisterAnimalUseCase e dipende solo da AnimalRepository (il port out) — mai dall'implementazione JPA concreta.
// application/RegisterAnimalService.java @ApplicationScoped public class RegisterAnimalService implements RegisterAnimalUseCase { private final AnimalRepository repository; // port, non JPA! public RegisterAnimalService(AnimalRepository repository) { this.repository = repository; } @Override @Transactional public Animal register(RegisterAnimalCommand cmd) { var animal = new Animal(UUID.randomUUID(), cmd.name(), ...); return repository.save(animal); } }
Quali use case implementare subito
Solo quelli strettamente necessari per una demo end-to-end. Gli altri li aggiungi dopo aver verificato il flusso completo.
RegisterAnimalService ← POST /animals GetAnimalService ← GET /animals/{id} ListAnimalsService ← GET /animals UpdateAnimalStatusService ← PATCH /animals/{id}/status
Test con Mockito — niente @QuarkusTest
I use case si testano mockando l'AnimalRepository. Nessun container Quarkus, nessun DB — test veloci e isolati.
// RegisterAnimalServiceTest.java @ExtendWith(MockitoExtension.class) class RegisterAnimalServiceTest { @Mock AnimalRepository repository; @InjectMocks RegisterAnimalService service; @Test void shouldRegisterAnimal() { ... } }
Infrastructure layer — adapter JPA e REST
Obiettivo: implementare i port "out" con JPA, esporre i port "in" via REST. Quarkus e Hibernate vivono solo qui.
JPA adapter — implementa AnimalRepository
La AnimalEntity ha le annotazioni JPA. L'adapter converte da domain model a entity e viceversa. Questa è l'unica classe che conosce Hibernate.
// infrastructure/persistence/AnimalRepositoryAdapter.java @ApplicationScoped public class AnimalRepositoryAdapter implements AnimalRepository { @Override public Animal save(Animal animal) { AnimalEntity entity = AnimalMapper.toEntity(animal); entity.persist(); return AnimalMapper.toDomain(entity); } }
Prima migration Flyway
Il file SQL va in src/main/resources/db/migration/. La naming convention di Flyway è obbligatoria: V{versione}__{descrizione}.sql
-- V1__create_animal_table.sql CREATE TABLE animal ( id UUID PRIMARY KEY, name VARCHAR(100) NOT NULL, species VARCHAR(100) NOT NULL, status VARCHAR(20) NOT NULL, arrival_date DATE NOT NULL );
REST resource — espone i use case
Il resource dipende dai port in, non dai service concreti. Converte HTTP → command → use case → response DTO. Non contiene business logic.
// infrastructure/rest/AnimalResource.java @Path("/animals") @Produces(APPLICATION_JSON) @Consumes(APPLICATION_JSON) public class AnimalResource { private final RegisterAnimalUseCase register; // port! private final GetAnimalUseCase getAnimal; @POST public Response register(RegisterAnimalRequest req) { Animal animal = register.register(toCommand(req)); return Response.created(...).entity(toResponse(animal)).build(); } }
✓ Checkpoint finale: POST /animals crea un record, GET /animals/{id} lo ritorna. Swagger UI funziona. I test passano. Hai il tuo primo servizio funzionante end-to-end.
